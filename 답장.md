# 검증 의견에 대한 답장 및 앞으로의 권고

작성 기준일: 2026-06-10  
전제: 이 문서는 의견·우선순위·수정 지시사항만 정리한다. 현재 코드·데이터·노트북·기능은 수정하지 않았다.

## 1. 결론

사용자가 직접 검증한 내용과 앞으로의 계획에 **대체로 동의한다.**  
특히 다음 판단이 정확하다.

> 현재 프로젝트의 가장 큰 위험은 구현량 부족이 아니라, 평가자가 지정된 진입점을 그대로 실행했을 때 정상 작동한다는 증거와 제출 마무리가 부족하다는 점이다.

다만 실행 계획은 일부 순서를 바꾸고, 몇몇 항목의 표현을 더 정확하게 해야 한다.

### 내 최종 판단

- 구현 자체의 잠재력은 높다.
- 지금 가장 먼저 해야 할 일은 기능 추가가 아니라 **평가 계약의 clean-run 성공 확보**다.
- Task 1 분류기와 Task 2 챗봇을 지금 시점에 억지로 통합하는 것은 권하지 않는다.
- stale ChromaDB 정합성 해결은 `chatbot.sh` 최종 검증보다 **먼저** 수행해야 한다.
- 미래 날짜 문제는 현재 코드가 일부 보완된 것이 맞지만, 출력 재생성 후 실제 해결 여부를 확인하기 전까지 완료로 판정하면 안 된다.

## 2. 사용자 검증 의견에서 동의하는 점

### 분류기 노트북

동의한다. 현재 분류기 노트북은 평가자가 바로 실행하기에 안전하지 않다.

다만 수정 방향은 단순히 경로를:

```text
/content/drive/MyDrive/CNU-QA-chatbot
```

에서:

```text
/content/drive/MyDrive/NLP_Term_Project
```

로 바꾸는 것이어서는 안 된다.

그렇게 하면 사용자 환경에서는 실행되지만, 조교가 ZIP을 다른 이름이나 위치에 풀었을 때 다시 실패한다. 구현 담당자에게는 **프로젝트 위치를 자동 탐색하거나 한 곳에서만 명시적으로 설정할 수 있는 위치 독립적 노트북으로 만들라**고 지시해야 한다.

### MOCK 출력

동의한다. `chat_output.json` 5건 전체가 MOCK이라는 사실은 현재 제출 상태의 신뢰도를 크게 낮춘다.

다만 이것만으로 현재 챗봇 코드가 반드시 고장났다고 단정할 수는 없다. 정확한 판정은 다음과 같다.

- 현재 저장된 출력은 실제 챗봇 정상 작동 증거가 아니다.
- 평가 진입점인 `chatbot.sh`의 clean-run 성공도 아직 증명되지 않았다.
- 따라서 최종 실행으로 실제 출력을 재생성하고 질문별 품질을 확인해야 한다.

### 벡터 DB 불일치

강하게 동의한다. 현재 가장 조용하면서 치명적인 위험이다.

- 현재 청크: 205개
- 현재 DB embeddings: 158개
- `chatbot.sh`는 `chroma_db` 폴더가 존재하면 재구축을 건너뜀
- 현재 임베더는 KURE-v1 1024차원을 기대함

이 상태에서 `chatbot.sh`부터 실행하는 것은 올바른 순서가 아니다. stale DB를 재사용해 실패하거나, 일부 데이터만 검색한 결과를 새 출력으로 남길 수 있다.

**DB 정합성 확보 후 `chatbot.sh`를 실행해야 한다.**

### 검증셋 누수

동의한다. `valid.json` 30개 중 25개가 학습셋과 정확히 겹치므로 현재 내부 F1은 일반화 성능 증거로 사용할 수 없다.

다만 마감 직전에 단순히 중복 25개를 삭제해서 valid를 5개만 남기는 것도 잘못된 대응이다. 구현 담당자에게는 다음 중 하나를 선택하도록 지시해야 한다.

1. 전체 데이터 중복 제거 후 라벨별로 분리된 새로운 stratified validation set 구축
2. 시간이 부족하면 현재 내부 F1을 발표에서 일반화 성능으로 주장하지 않고, 조교 비공개 테스트셋 결과를 최종 기준으로 설명

가능하면 1번이 좋지만, 평가 진입점 실행 안정성보다 먼저 할 일은 아니다.

## 3. 의견을 조정해야 하는 부분

### 미래 날짜 문제는 “해결됨”이 아니라 “재검증 대기”로 판정해야 함

사용자의 뉘앙스 지적은 타당하다.

- `outputs/realtime_output.json` 생성 시각: 2026-06-09 03:02
- `notice_handler.py` 수정 시각: 2026-06-09 19:22
- 현재 `_extract_date()`의 일반 ISO/점 형식 날짜 경로에는 미래 날짜 제외 로직이 존재한다.
- 기존 `2026-12-31`은 본문의 이수기한을 게시일처럼 읽은 결과이며, 현재 일반 날짜 필터라면 제외될 가능성이 높다.

그러나 완전히 해결됐다고 단정하면 안 된다.

- `등록일` 패턴과 `접수기간` 패턴은 발견 즉시 반환하며 미래 여부를 별도로 검사하지 않는다.
- 현재 출력 파일은 수정 이후 재생성되지 않았다.
- “가장 최근 공지” 질문을 다시 실행한 결과가 아직 없다.

따라서 이 항목은 **현재 코드 버그 확정**에서 **stale 출력 가능성이 높은 재검증 필수 항목**으로 강등하는 것이 적절하다.

완료 조건은 단순 출력 재생성이 아니라 다음 세 가지다.

1. 정확히 같은 질문을 다시 실행
2. 답변에 오늘 이후 날짜가 “게시일”로 나오지 않는지 확인
3. 출력된 날짜가 실제 게시일인지 원문 링크로 확인

### Task 1 분류기와 Task 2 라우터 미연결

사용자 의견에 동의한다. 현재 마감 상황에서는 차단급으로 볼 필요가 없다.

과제 문구상 분류 결과 기반 응답 생성으로 해석할 여지는 있지만, 평가 진입점과 배점은 Task 1과 Task 2를 별도로 평가한다. 지금 분류기를 챗봇에 통합하면 다음 위험이 생긴다.

- 챗봇 실행 시 분류기 모델 로딩 시간 및 의존성 증가
- 새로운 라우팅 회귀 발생
- 이미 동작하는 식단·셔틀·공지 전용 핸들러 경로가 깨질 가능성

따라서 현재는 코드를 통합하기보다 발표에서 다음처럼 설명하는 편이 낫다.

> Task 1은 독립적인 5분류 성능 평가를 위한 학습 모델이며, Task 2는 응답 안정성과 실시간 핸들러 활용을 위해 규칙 기반 고속 라우팅과 RAG를 결합했다.

이 항목은 **발표 설명 및 설계 선택 정당화 사항**으로 다루는 것을 권한다.

## 4. 계획에서 반드시 바꿔야 할 실행 순서

사용자 계획의 가장 중요한 수정점은 `chatbot.sh` 검증보다 DB 정합성 확보를 먼저 수행하는 것이다.

## 권장 실행 순서

### Phase A. 평가 진입점 자체를 실행 가능하게 만들기

1. `classifier_박연진.ipynb`를 위치 독립적으로 만들기
2. 새 Colab T4 런타임에서 셀을 처음부터 끝까지 실행
3. 실제 macro F1, `cls_output.json`, 실행 시간 기록

### Phase B. 챗봇 검색 기반 정합성 확보

4. 기존 DB가 현재 임베더·현재 청크와 맞는지 검증하는 조건 마련
5. 205개 청크 기준으로 DB를 새로 구축
6. embeddings 수 205개 및 임베딩 차원 1024 확인
7. 대표 검색 질문으로 dense/BM25/RRF 결과 확인

### Phase C. 평가 명령 clean-run

8. 완전히 새 Colab T4 런타임에서 `bash chatbot.sh` 실행
9. `chat_output.json`에 MOCK이 없고 입력 개수와 출력 개수가 같은지 확인
10. UI 접속, 질문 입력, 응답 출력, 대화 흐름 표시 확인
11. `realtime_output.json` 재생성 후 미래 게시일·카테고리·셔틀 서식 확인

### Phase D. 제출 일관성 및 품질

12. README의 7B 설명을 실제 기본 3B와 일치시키기
13. 모델 파일 또는 다운로드 링크와 권한 확인
14. 검증셋 누수 처리 또는 발표에서 내부 F1의 한계 명시
15. 발표자료와 UI 영상 제작
16. 최종 제출 전용 디렉터리에서 ZIP 생성 및 다시 압축 해제하여 실행 파일 확인

## 5. 구현 담당자에게 전달해야 할 구체적인 수정 지시

아래 내용은 그대로 구현 작업 지시서로 사용할 수 있다.

### P0: 제출 전 반드시 수정

#### 1. 분류기 노트북 위치 독립성

- 특정 Google Drive 폴더명을 하드코딩하지 말 것.
- 조교가 다른 폴더명에 배치해도 실행 가능하게 만들 것.
- 새 런타임에서 셀 순서대로 실행 시 수동 코드 수정이 없어야 함.
- `data/test_cls.json` 교체 후 `outputs/cls_output.json`을 자동 생성해야 함.
- 성공 증거로 macro F1과 출력 JSON을 저장할 것.

#### 2. 벡터 DB 재사용 조건

- `chroma_db` 폴더 존재 여부만으로 재사용하지 말 것.
- 최소한 다음이 모두 맞을 때만 기존 DB를 사용하도록 할 것.
  - 현재 `chunks.json` ID/개수와 DB ID/개수 일치
  - 현재 임베딩 모델과 DB 생성 모델 일치
  - 임베딩 차원 일치
- 불일치 시 DB를 재구축하도록 할 것.
- 최종 검증에서 `chunks=205`, `embeddings=205`, `dimension=1024`를 증명할 것.

#### 3. 평가 명령 종단 실행

- 새 Colab T4 환경에서 `bash chatbot.sh` 한 명령으로 다음이 모두 완료되어야 함.
  - 실제 `chat_output.json` 생성
  - Optional 입력이 있으면 `realtime_output.json` 생성
  - Gradio UI 정상 실행
  - 질문 입력 및 응답 확인
- MOCK 결과가 생성되거나 기존 MOCK 파일이 최종 제출물에 남지 않도록 할 것.

#### 4. 미래 날짜 재검증

- 수정된 코드로 `realtime_output.json`을 재생성할 것.
- “가장 최근에 올라온 공지사항은 언제 게시되었나요?”를 반드시 포함할 것.
- 오늘 이후 날짜를 게시일로 반환하지 않는지 검사할 것.
- `등록일`, `접수기간`, 일반 본문 날짜 각각의 처리 기준을 확인할 것.

### P1: 높은 점수를 위해 수정

#### 5. 문서와 실제 모델 일치

- README의 기본 모델 설명을 실제 실행 기본값인 Qwen2.5-3B와 일치시킬 것.
- 7B는 실험용으로 명확히 구분할 것.
- 첫 실행 예상 시간, 다운로드 용량, GPU 요구사항을 실제 clean-run 결과 기준으로 작성할 것.

#### 6. 검증셋 누수

- train/valid 정확 중복을 제거할 것.
- 라벨별 분포를 유지한 새 validation set을 만들 것.
- 난수 시드를 고정해 재학습 결과의 변동을 줄일 것.
- 수정 후 macro F1을 다시 측정하되, 점수가 내려가도 그것이 더 신뢰할 수 있는 결과임을 이해할 것.

#### 7. 제출 패키지

- 최종 ZIP에는 평가와 실행에 필요한 파일만 포함할 것.
- `.git`, `.omc`, `.omx`, `__pycache__`, 중복/실험 노트북, 오래된 MOCK 출력은 제외할 것.
- 발표자료, UI 영상, 모델 파일 또는 다운로드 링크를 포함할 것.
- 공유 링크는 다른 계정 또는 로그아웃 상태에서 다운로드 가능 여부를 확인할 것.

### P2: 현재 마감에서는 설명으로 처리 가능

- Task 1 분류기와 Task 2 키워드 라우터 통합

## 6. 각 단계의 완료 판정 기준

“실행해 봤다”가 아니라 아래 증거가 있어야 완료다.

| 단계 | 완료 증거 |
|---|---|
| 분류기 | 새 런타임 전체 실행 성공, macro F1 출력, 입력/출력 개수 일치 |
| DB | chunks 수 = embeddings 수 = 205, 차원 1024, 대표 검색 성공 |
| 챗봇 | 새 런타임 `bash chatbot.sh` 성공, MOCK 없음, UI 질문/응답 성공 |
| 실시간 | 동일 질문 재실행 시 미래 게시일 없음, 링크 원문과 날짜 일치 |
| 제출물 | 발표자료·2분 영상·모델/링크·최종 ZIP 존재 |
| 최종 패키지 | ZIP을 별도 위치에 풀어 README 절차대로 clean-run 가능 |

## 7. 시간 부족 시 포기해도 되는 것과 포기하면 안 되는 것

### 절대 포기하면 안 됨

- 분류기 노트북의 수동 경로 수정 요구 제거
- DB 청크/임베딩/차원 정합성
- `chatbot.sh` clean-run
- MOCK 없는 실제 출력
- 발표자료, UI 영상, 최종 ZIP

### 시간이 부족하면 설명으로 대체 가능

- Task 1 분류기를 Task 2 라우팅에 직접 연결
- API와 UI 동작 완전 통일
- 대규모 데이터 추가 수집
- UI 디자인 추가 개선
- 새로운 기능 추가

## 8. 최종 의견

사용자의 계획은 방향이 맞고, 핵심 사실 검증도 신뢰할 만하다.  
다만 성공 가능성을 높이려면 다음 세 문장을 기준으로 움직여야 한다.

1. **기능을 더 만들지 말고 평가자가 실행하는 두 진입점을 먼저 완성한다.**
2. **stale DB를 해결한 뒤에만 `chatbot.sh` 최종 성공을 판정한다.**
3. **수정 완료 주장이 아니라 새 환경 clean-run 증거로 완료를 판정한다.**

Phase A~C가 끝나기 전에는 발표자료 꾸미기나 추가 기능 개발에 시간을 사용하지 않는 것이 좋다. 현재 프로젝트는 새 기능이 필요한 상태가 아니라, 이미 만든 기능을 평가자가 확실히 실행할 수 있도록 고정해야 하는 상태다.
