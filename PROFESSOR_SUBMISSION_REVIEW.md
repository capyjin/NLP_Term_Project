## 1. 총평

프로젝트의 기술적 범위와 구현량은 충분히 좋다. 5분류기, RAG, 생성 모델, 웹 UI, 실시간 데이터 핸들러, API까지 과제 기본 범위를 넘어 구현했다. 특히 평가용 진입점인 `classifier_박연진.ipynb`와 `chatbot.sh`, 요구 출력 JSON 구조를 모두 준비한 점은 강점이다.

그러나 **현재 상태 그대로 제출하면 높은 점수를 장담할 수 없다.** 가장 큰 문제는 기능 부족이 아니라 **평가자가 그대로 실행할 때의 재현성 및 제출 완성도**다. 분류기 노트북 경로가 수정 필요 상태이고, 현재 `chat_output.json`은 실제 답변이 아닌 `[MOCK]`이며, 발표자료·UI 영상·최종 ZIP이 없다. 기존 Colab 실행 기록에도 UI 오류와 패키지 충돌 경고가 남아 있다.

### 교수 관점 판정

- 코드/구현 잠재력: **우수**
- 현재 제출 완성도: **미완성**
- 현재 상태 최종 제출 권고: **REQUEST CHANGES**
- 아키텍처 상태: **BLOCK**
- 핵심 이유: 평가 진입점 재현성, 실제 출력 증거, 필수 제출물 누락

## 6. 잘못된 제출로 이어질 수 있는 주의사항

- 조교가 실행하는 것은 사실상 두 진입점뿐이다. 다른 스크립트가 잘 작동해도 `classifier_박연진.ipynb` 또는 `chatbot.sh`가 실패하면 점수를 잃는다.
- 현재 샘플 `data/test_*.json`만 맞추는 것은 충분하지 않다. 조교가 파일을 교체해도 일반화되어야 한다.
- 검증셋이 학습셋과 중복되므로 현재 내부 F1을 실제 성능으로 제시하면 안 된다.
- `chatbot.sh`는 네트워크, Hugging Face 다운로드, 크롤링, 벡터 DB, LLM, UI를 한 번에 수행한다. 어느 한 단계의 실패도 평가 실패로 연결될 수 있다.
- Optional 실시간 기능은 “있음”보다 “정확함”이 중요하다. 미래 날짜를 최신 공지로 답하는 결과는 기능이 없는 것보다 더 나쁜 인상을 줄 수 있다.
- 발표에서 7B 모델을 사용한다고 설명하면 실제 3B 기본 코드와 충돌한다.
- 공유 링크를 제출할 경우 반드시 조교 계정에서 다운로드 가능한 권한인지 확인해야 한다.

## 7. 조건부 점수 예상

### 현재 상태 그대로 제출할 경우

| 항목 | 예상 | 이유 |
|---|---:|---|
| Task 1 분류기 | 0~35 / 40 | 구현은 좋으나 Colab 경로 및 실행 증거 부족으로 실패 가능 |
| Task 2 UI/Chat | 15~40 / 60 | UI 구현은 있으나 종단 실행 오류 위험 및 현재 MOCK 결과 |
| Optional 실시간 | 5~15 / 30 | 기능은 많으나 최신성·정확성 오류 증거 |
| **예상 총점** | **20~90 / 130** | 실행 성공 여부에 따라 변동 폭이 매우 큼 |

### 모든 필수 검증과 제출물이 갖춰진 경우의 잠재력

기술적 구현 범위를 고려하면 **100점 이상/130점**도 가능하다. 현재 점수 위험의 대부분은 구현량이 아니라 재현성, 정확성 검증, 제출 패키징에서 발생한다.
